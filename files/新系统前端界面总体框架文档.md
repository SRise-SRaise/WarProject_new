# 新系统前端逻辑与界面框架设计文档

## 1. 文档定位

### 1.1 文档目标

本文档用于定义新系统前端的逻辑结构与界面框架，作为后续页面设计、路由组织、模块拆分、状态管理和前后端联调的统一依据。

当前项目不是在旧系统页面上做局部修补，而是在新技术架构下对旧业务进行重新组织。因此，本文档不再沿用“旧系统菜单罗列”的写法，而是以当前仓库已经具备的前端骨架为基础，明确新系统前端应该如何落地。

### 1.2 文档范围

本文档重点覆盖以下内容：

- 当前前端项目的真实基线，包括布局、路由、状态、请求边界和模块目录
- 新系统前端的整体信息架构和模块归属逻辑
- 学生端与教师端的页面承载方式、入口关系和任务链路
- 前端路由与壳层的组织方式
- 业务页面应如何落到 `user`、`homework`、`exam`、`experiment`、`common` 模块
- 前端状态管理、接口调用、交互协同和页面刷新规则
- 第一阶段应优先落地的页面矩阵

### 1.3 不在本文档范围内的内容

以下内容不在本文档的主范围内：

- 视觉风格、品牌色、插图和装饰性布局
- 单个组件的像素级样式设计
- 接口字段级设计和数据库结构改造
- 后端部署架构和中间件选型

### 1.4 设计表达规则

为避免把旧系统经验直接套到新项目上，本文档遵循以下表达规则：

- 凡是当前仓库已经存在的结构，按“当前基线”描述
- 凡是为了新系统业务落地而新增的组织方式，按“目标设计”描述
- 角色是导航与权限维度，不再作为前端代码目录维度
- 代码目录维度以当前项目已确定的业务模块为准：`user`、`homework`、`exam`、`experiment`、`common`

## 2. 当前项目前端基线

### 2.1 应用装配基线

当前前端项目采用 Vue 3 + Vite + TypeScript + Pinia + Ant Design Vue 的组合，应用入口已完成以下基础装配：

- `main.ts` 已注册 Pinia、Vue Router 和 Ant Design Vue
- 全局样式由 `src/styles/theme.css` 提供基础色彩变量
- 全局请求入口为 `src/request.ts`
- 接口层已存在 OpenAPI 生成产物 `src/api/*`

这意味着新系统前端不是从零起步，而是在一个已经具备“应用装配、布局骨架、路由聚合、请求层”的模板型工程上继续向业务系统演进。

### 2.2 目录与模块基线

当前 `src/` 目录已经体现出明确的前端分层：

| 目录 | 当前作用 | 说明 |
| --- | --- | --- |
| `src/layouts/` | 壳层布局 | 已有 `BasicLayout` 和 `BackendLayout` |
| `src/router/` | 路由定义 | `index.ts` 自动聚合所有路由文件 |
| `src/views/` | 页面视图 | 当前存在模板页和业务模块预留目录 |
| `src/components/` | 复用组件 | 当前已有 dashboard 组件和模块预留目录 |
| `src/stores/` | 状态管理 | 当前只有 demo store，业务 store 尚未落地 |
| `src/api/` | 接口函数 | 由 OpenAPI 生成，不应手写业务接口实现 |
| `src/utils/` | 工具能力 | 放通用工具，不承载业务状态 |

当前项目已经建立了模块目录骨架：

- `views/user`、`views/homework`、`views/exam`、`views/experiment`
- `components/user`、`components/homework`、`components/exam`、`components/experiment`
- `stores/user`、`stores/homework`、`stores/exam`、`stores/experiment`
- `common` 相关公共视图、组件和 store 目录也已预留

这说明新系统前端的正确方向不是继续堆页面，而是按既有模块骨架把业务落位。

### 2.3 路由与标题机制基线

当前路由系统已经具备两个关键特征：

#### 2.3.1 路由自动聚合

`src/router/index.ts` 通过 `import.meta.glob('./*.ts', { eager: true })` 自动收集所有路由文件，因此后续业务路由应继续按模块拆到独立文件中，而不是回退到一个超长的集中式路由文件。

#### 2.3.2 标题与导航依赖路由元数据

- `src/router/guards.ts` 使用 `meta.title` 统一设置页面标题
- `BackendLayout.vue` 使用子路由的 `meta.title`、`meta.icon`、`meta.group` 动态构建后台菜单

因此，前端菜单结构不应再额外维护一套独立的菜单配置树，而应由路由元数据驱动。

### 2.4 当前布局基线

当前项目已有两个布局壳层：

| 布局 | 当前用途 | 目标用途 |
| --- | --- | --- |
| `BasicLayout` | 公共首页、登录、注册、404 等基础页面承载 | 第一阶段继续承载公共页和学生端页面 |
| `BackendLayout` | `/admin` 下的后台模板页面承载 | 第一阶段作为教师端壳层 |

这意味着：

- 第一阶段无需新增第三个壳层即可开始业务落地
- 学生端可先复用 `BasicLayout`
- 教师端统一挂到 `BackendLayout` 下，由后台路由元数据生成菜单

后续如果学生端规模扩大，再单独拆分 `StudentLayout` 或 `AuthLayout`，但这属于第二阶段重构，不作为当前文档的前置条件。

### 2.5 请求与 API 基线

当前请求层已经建立统一入口：

- `src/request.ts` 统一配置 `baseURL`、超时和 `withCredentials`
- 环境变量通过 `VITE_API_BASE_URL` 控制请求前缀
- Vite 本地代理将 `/api` 转发到后端服务
- `src/api/*` 当前已存在部分 OpenAPI 生成产物，应视为接口访问边界

这意味着后续业务页面不应自己拼接 URL，也不应在 `src/api/` 中手写额外业务实现。当前业务接口尚未完整暴露，后续页面的数据访问应建立在“后端暴露业务接口 -> 重新生成客户端 -> 统一请求层接入 -> store 或页面逻辑消费”的链路之上。

### 2.6 当前脚手架缺口

当前前端还处于“模板已成型、业务尚未补全”的状态，主要缺口如下：

- 登录页仍是示例逻辑，提交后直接跳到 `/admin/dashboard`
- 当前没有真实登录态存储和权限路由拦截
- `admin.ts` 目前只有工作台一条业务路由
- `user.ts`、`homework.ts`、`exam.ts`、`experiment.ts`、`common.ts` 还是空壳
- `NavBar` 仍是模板型导航，并未体现学生端真实业务入口
- `counter.ts` 仍是 demo store，业务 store 尚未建立
- `HomeView`、`DashboardView` 仍带有明显模板文案

因此，本文档的目标不是描述“已经完成了什么”，而是明确“在现有前端骨架上如何把新系统做实”。


## 3. 新系统前端总体逻辑设计

### 3.1 总体结构定义

新系统前端采用“三段式逻辑”：

1. 公共入口层：承载首页、登录、注册、异常页等公共能力
2. 学生学习层：承载学生的学习任务、资料查看、作业、实验、考试和个人信息
3. 教师管理层：承载教师的教学组织、资源维护、任务发布、过程跟踪和结果处理

这里的“学生端”和“教师端”是产品使用视角；在代码组织上，不再建立 `student` 和 `teacher` 两套并行目录，而是统一收敛到业务模块。

### 3.2 模块归属原则

新系统前端的模块归属应以当前项目和后端控制器命名为依据：

| 业务能力 | 前端模块归属 | 归属说明 |
| --- | --- | --- |
| 登录、注册、个人信息、密码修改、班级、学生、教师账户 | `user` | 与身份、账号、教学对象相关的能力统一归到 `user` |
| 学习概览、教师工作台、课程资料、公共配置、公共页 | `common` | 与单一业务域无强绑定，但全端通用或作为公共入口的能力归到 `common` |
| 作业、作业题目、作业提交、提交记录、批改 | `homework` | 所有作业过程页和结果页都归到 `homework` |
| 题库、题型、试卷、考试、考试记录、成绩汇总 | `exam` | 题库维护入口仍归 `exam` 模块，但作为作业与考试共用的题目资产使用 |
| 实验、实验项、实验结果、实验报告查看与处理 | `experiment` | 实验结果不再独立成顶层模块，而是实验域内部子流程 |

### 3.3 角色与模块关系

角色维度和模块维度的关系应明确区分：

- 模块决定代码落位
- 角色决定页面可见性、操作范围和默认入口

例如：

- `homework` 模块既有学生页，也有教师页
- 学生看到的是“作业列表、作业详情、提交结果”
- 教师看到的是“作业列表、作业编辑、发布范围、提交记录、批改结果”

也就是说，同一模块按角色呈现不同页面链路，而不是拆出两套重复前端结构。

### 3.4 页面层级标准

业务模块优先参考以下页面层级模式：

| 页面层级 | 学生端含义 | 教师端含义 |
| --- | --- | --- |
| 概览页 | 展示待办、最近记录、快捷入口 | 展示待处理事项、统计概览、快捷入口 |
| 列表页 | 展示可参与任务或可查看资源 | 展示可维护对象或可处理任务 |
| 详情页 | 展示单个任务或资源说明 | 展示单个对象、任务或配置详情 |
| 处理页 | 提交、作答、下载、查看结果 | 新建、编辑、发布、批改、统计 |
| 结果页 | 展示个人结果、状态、反馈 | 展示班级结果、提交记录、评分汇总 |

设计重点不是增加多少页面，而是保证主要业务模块具备清晰的“概览 -> 列表 -> 详情 -> 处理/结果”链路；不要求每个模块都机械补齐五层。

## 4. 路由与壳层设计

### 4.1 路由文件职责划分

目标设计中，各路由文件承担如下职责：

| 路由文件 | 承担内容 | 布局 |
| --- | --- | --- |
| `basic.ts` | 公共首页、登录、注册、404、无权限等公共页 | `BasicLayout` |
| `admin.ts` | 教师端总入口 `/admin`、工作台、后台菜单汇总 | `BackendLayout` |
| `user.ts` | 学生个人中心、教师用户管理相关页面定义 | 学生页走 `BasicLayout`，教师页作为 `/admin` 子路由 |
| `homework.ts` | 学生作业链路和教师作业管理链路 | 学生页走 `BasicLayout`，教师页作为 `/admin` 子路由 |
| `exam.ts` | 学生考试链路和教师题库/试卷/考试管理链路 | 学生页走 `BasicLayout`，教师页作为 `/admin` 子路由 |
| `experiment.ts` | 学生实验链路和教师实验管理链路 | 学生页走 `BasicLayout`，教师页作为 `/admin` 子路由 |
| `common.ts` | 学习概览、课程资料、教师工作台之外的公共业务页 | 按角色分别挂入对应壳层 |

### 4.2 路由组合规则

为了同时满足“按模块拆文件”和“教师端统一使用 `/admin` 壳层”两个目标，建议采用以下组合规则：

- `index.ts` 继续负责自动聚合所有默认导出的顶层路由
- 学生侧或公共页可直接由各模块文件默认导出顶层路由
- 教师侧业务路由可以按模块分别定义，但最终都必须挂载为 `/admin` 的子路由，由 `admin.ts` 汇总或组合这些后台子路由定义

这样可以保持：

- 路由按模块拆分
- 教师端始终只有一个后台壳层
- `BackendLayout` 的菜单来源仍然唯一且稳定

### 4.3 壳层职责定义

#### 4.3.1 `BasicLayout`

第一阶段 `BasicLayout` 同时承担两类职责：

- 公共页承载：首页、登录、注册、异常页
- 学生端承载：学习概览、课程资料、作业、实验、考试、个人中心

因此，`BasicLayout` 的导航逻辑需要从当前模板型 `NavBar` 演进为“公共态导航 + 学生态导航”的组合模式：

- 未登录时显示公共入口
- 学生登录后显示学习相关入口和个人入口
- 不在 `BasicLayout` 中暴露教师端后台菜单

#### 4.3.2 `BackendLayout`

`BackendLayout` 继续作为教师端唯一业务壳层，承担：

- 后台头部信息区
- 左侧菜单导航
- 当前页内容承载
- 路由驱动的菜单高亮和菜单分组

教师端的菜单不再手工维护一份平铺导航表，而是直接依赖后台子路由的 `meta.title`、`meta.icon`、`meta.group`。

### 4.4 路由元数据约束

所有业务路由都应携带清晰的路由元数据，其中至少应保证：

- `name`：唯一且稳定，用于路由跳转和菜单高亮
- `meta.title`：页面标题来源，也是后台菜单标题来源

对于挂载到教师端后台菜单中的路由，再按需补充：

- `meta.icon`：教师端顶级菜单图标来源
- `meta.group`：教师端菜单分组来源

页面标题和后台菜单仍然统一依赖路由元数据，而不是复制出另一套配置源。

### 4.5 登录与分流逻辑

当前登录页仍是模板行为，新系统应调整为“后端返回身份，前端负责分流”的模式。

目标逻辑如下：

1. 登录页保留账号和密码输入，不再把角色选择当作前端长期方案
2. 登录成功后，由登录接口返回当前用户身份和权限范围
3. 前端将用户信息写入 `user` 模块的登录态 store
4. 前端根据角色跳转：
   - 教师进入 `/admin/dashboard`
   - 学生进入学生端学习概览页
5. 未登录访问受保护页面时，统一重定向到登录页

这样做的好处是：

- 角色由后端身份决定，而不是登录页表单决定
- 一个账号只走一条明确业务链路
- 前端不需要同时维护“登录页角色选择逻辑”和“后台权限逻辑”两套规则

### 4.6 注册逻辑处理原则

当前项目保留了注册页，但注册页是否长期对外开放应作为实施开关处理：

- 如果后端允许学生自注册，则注册页保留在公共入口层
- 如果账号由教师或系统统一分配，则注册页可下线或仅作为预留页

因此，注册页不应主导整个信息架构，只应作为 `user` 模块下的一个可开关能力。


## 5. 学生端逻辑设计

### 5.1 学生端定位

学生端的目标不是展示所有系统能力，而是围绕“查看任务、完成任务、查看结果、维护个人信息”形成稳定学习闭环。

学生端页面应围绕以下五类入口组织：

- 学习概览
- 课程资料
- 作业
- 实验
- 考试
- 个人中心

### 5.2 学生端模块与页面链路

以下表中的视图目录和 store 文件名属于建议命名，用于指导后续落位，不表示这些文件当前已经存在。

同时，学习概览与课程资料归入 `common` 是前端组织选择，用于承载公共学习入口和教学资源，不表示后端当前已经提供对应的聚合型概览接口。

| 学生能力 | 模块归属 | 路由文件 | 布局 | 建议视图目录 | 页面链路 | 建议状态归属 |
| --- | --- | --- | --- | --- | --- | --- |
| 学习概览 | `common` | `common.ts` | `BasicLayout` | `views/common/student/` | 概览页 -> 最近任务 -> 对应模块详情 | `stores/common/app.ts` |
| 课程资料 | `common` | `common.ts` | `BasicLayout` | `views/common/material/` | 列表页 -> 详情页/下载 | 页面局部状态 + `stores/common/material.ts` |
| 作业 | `homework` | `homework.ts` | `BasicLayout` | `views/homework/student/` | 列表页 -> 详情页 -> 提交页 -> 结果页 | `stores/homework/student.ts` |
| 实验 | `experiment` | `experiment.ts` | `BasicLayout` | `views/experiment/student/` | 列表页 -> 详情页 -> 作答页/结果页 | `stores/experiment/student.ts` |
| 考试 | `exam` | `exam.ts` | `BasicLayout` | `views/exam/student/` | 列表页 -> 进入确认页 -> 答题页 -> 结果页 | `stores/exam/student.ts` |
| 个人中心 | `user` | `user.ts` | `BasicLayout` | `views/user/student/` | 个人信息页 -> 修改密码页 | `stores/user/auth.ts` |

### 5.3 学生端首页逻辑

学生端首页不应承担完整业务处理，只承担“学习概览”和“任务分发”职责，建议包含以下区块：

- 待完成任务：作业、实验、考试的待办列表
- 最近结果：最近一次提交、考试成绩、实验结果摘要
- 快捷入口：进入资料、作业、实验、考试
- 个人提示：账号状态、通知提示、密码修改提醒

首页只做任务聚合，不替代业务模块的列表页。

### 5.4 课程资料逻辑

课程资料建议归入 `common` 模块，而不是额外创建 `material` 模块，原因是当前后端资料能力位于公共控制器中，且资料本身更像教学资源供给能力。

学生侧资料页逻辑应包含：

- 列表筛选：课程、类型、关键词
- 资源详情：资料说明、上传时间、可见范围、附件信息
- 行为操作：查看、下载、返回资料列表

资料页不应混入提交、批改、成绩等流程型逻辑。

### 5.5 作业逻辑

学生作业链路建议明确为：

- 作业列表页：展示状态、截止时间、所属课程
- 作业详情页：展示要求、附件、提交规则
- 作业提交页：上传附件、填写内容、提交确认
- 作业结果页：展示提交状态、教师反馈、得分结果

其中“结果页”属于作业域内部页面，不应单独升级为学生端一级导航。

### 5.6 实验逻辑

学生实验链路建议明确为：

- 实验列表页：展示可参与实验、状态、开放时间
- 实验详情页：展示实验目标、实验项、说明材料
- 实验处理页：执行实验步骤、填写结果、上传报告
- 实验结果页：查看实验成绩、反馈或结果记录

实验报告相关内容对于学生而言是实验流程的一部分，不应再拆成独立导航模块。

### 5.7 考试逻辑

学生考试链路建议明确为：

- 考试列表页：展示待参加、已完成、未开放考试
- 考试进入确认页：展示考试说明、时长、规则
- 答题页：承载真实作答逻辑
- 结果页：展示考试状态、成绩或结果摘要

考试结果属于 `exam` 模块内部结果链路，不应与“题库”“试卷”等教师能力混合描述。

## 6. 教师端逻辑设计

### 6.1 教师端定位

教师端承载的是“组织教学对象、维护教学资源、发起教学活动、处理教学结果”的完整管理链路。教师端不再按旧系统菜单平铺，而是按业务模块稳定组织。

### 6.2 教师端模块与页面链路

以下表中的视图目录和 store 文件名属于建议命名，用于指导后续落位，不表示这些文件当前已经存在。

其中，工作台与课程资料归入 `common` 也是前端页面组织选择，用于承载公共后台入口和教学资源管理，不表示后端当前已经存在对应的聚合型后台工作台接口。

| 教师能力 | 模块归属 | 路由文件 | 布局 | 建议视图目录 | 页面链路 | 建议状态归属 |
| --- | --- | --- | --- | --- | --- | --- |
| 工作台 | `common` | `admin.ts` 或 `common.ts` | `BackendLayout` | `views/common/admin/` | 工作台 -> 模块快捷入口 -> 待办详情 | `stores/common/app.ts` |
| 用户管理 | `user` | `user.ts` | `BackendLayout` | `views/user/admin/` | 列表页 -> 详情页 -> 编辑页 | `stores/user/admin.ts` |
| 课程资料 | `common` | `common.ts` | `BackendLayout` | `views/common/material-admin/` | 列表页 -> 编辑页 -> 发布状态页 | `stores/common/material.ts` |
| 作业管理 | `homework` | `homework.ts` | `BackendLayout` | `views/homework/admin/` | 列表页 -> 编辑页 -> 发布页 -> 提交记录/批改页 | `stores/homework/admin.ts` |
| 考试管理 | `exam` | `exam.ts` | `BackendLayout` | `views/exam/admin/` | 题库页 -> 试卷页 -> 考试页 -> 记录/成绩页 | `stores/exam/admin.ts` |
| 实验管理 | `experiment` | `experiment.ts` | `BackendLayout` | `views/experiment/admin/` | 列表页 -> 编辑页 -> 实验项管理页 -> 结果处理页 | `stores/experiment/admin.ts` |
| 个人设置 | `user` | `user.ts` | `BackendLayout` | `views/user/admin/` | 个人信息页 -> 修改密码页 | `stores/user/auth.ts` |

### 6.3 教师工作台逻辑

教师工作台不应继续停留在模板统计图层面，而应改造成教学任务入口页，建议包含：

- 待处理作业提交数
- 待批改实验结果数
- 近期考试安排和成绩汇总入口
- 快捷进入班级管理、学生管理、课程资料、作业、实验、考试

工作台的作用是“汇总和分发任务”，不是承载所有管理能力本身。

### 6.4 用户管理逻辑

教师端用户管理建议统一归入 `user` 模块，包括：

- 班级列表与班级详情
- 学生列表与学生详情
- 必要时的教师账号信息维护

“班级管理”和“学生管理”属于教学对象管理，应作为教学活动的上游基础能力保留。

### 6.5 课程资料逻辑

课程资料建议作为 `common` 模块下的公共教学资源能力处理，而不是再单独造一个前端模块。教师侧资料页应包含：

- 资料列表页：查看已有资料和可见范围
- 资料编辑页：新增、编辑、上传、下线
- 资料发布信息：课程归属、班级范围、附件状态

这样学生和教师都能围绕同一资源域使用不同入口。

### 6.6 作业逻辑

教师作业链路建议明确为：

- 作业列表页：查看作业状态、发布时间、覆盖范围
- 作业编辑页：维护作业基本信息和题目结构，支持手工组题或从共享题库中选题
- 作业发布页：选择班级、截止时间、提交规则
- 提交记录页：查看学生提交情况
- 批改页：处理评分、反馈和结果状态

这里的“提交记录”和“批改”都属于 `homework` 模块内部流程，不再提升为顶层模块。题库虽然被作业使用，但维护入口仍放在 `exam` 模块中，由作业模块按业务需要进行消费。

### 6.7 考试逻辑

教师考试模块及其共享题库资产建议按业务关系组织为：

- 题库管理
- 题型管理
- 试卷管理
- 考试管理
- 考试记录与成绩汇总

这几类能力都属于 `exam` 域的内部链路，因此：

- 题库不单列为前端一级模块
- 题库管理入口统一放在 `exam` 模块中，但同时服务作业与考试两条业务线
- 成绩汇总不单列为前端一级模块
- 所有考试结果页都作为考试域内部结果页组织

### 6.8 实验逻辑

教师实验链路建议明确为：

- 实验列表页：展示实验任务
- 实验编辑页：编辑实验基本信息
- 实验项管理页：管理实验步骤或实验项内容
- 实验结果页：查看学生结果、报告、批改情况

旧系统中的“实验报告”在新系统中不再作为顶层模块，而是实验域的结果处理页。


## 7. 状态、接口与交互规则

### 7.1 状态管理分层

新系统前端状态应按“全局状态、模块状态、页面局部状态”三层组织：

| 状态层级 | 建议归属 | 主要内容 |
| --- | --- | --- |
| 全局状态 | `stores/user/auth.ts`、`stores/common/app.ts` | 当前用户、角色、登录态、全局加载态、壳层折叠状态、全局通知 |
| 模块状态 | `stores/{module}/` | 列表缓存、查询条件、当前选中对象、跨页复用的数据 |
| 页面局部状态 | 视图组件内部 | 表单草稿、弹窗开关、临时筛选项、分页状态 |

原则上：

- 只要状态跨页面复用，就进入 store
- 只在当前页面使用的状态，保留在页面内部
- 不把所有表单字段和弹窗状态一股脑塞进全局 store

### 7.2 数据流规则

前端统一采用如下数据流：

`视图组件 -> store 或页面逻辑 -> src/api 生成函数 -> src/request.ts -> 后端接口`

使用原则如下：

- 页面禁止手写 URL 和 HTTP 请求细节
- 业务接口统一通过 `src/api/*` 调用
- 跨页面共享的数据获取、缓存和刷新放入 store
- 只在当前页使用的一次性请求逻辑可以留在页面的组合式逻辑中

### 7.3 登录态与权限规则

以下为目标登录态设计建议，当前仓库尚未实现完整鉴权守卫：

- `stores/user/auth.ts` 负责保存当前用户、角色、是否登录
- 路由守卫负责判断页面是否需要登录
- 教师端后台菜单只根据教师可见路由渲染
- 学生端不进入 `/admin` 命名空间
- 未登录访问受保护页面时统一回登录页

### 7.4 页面刷新与协同规则

新系统前端不建议引入全局业务 event bus。跨页面或跨模块的协同优先采用以下方式：

- 提交成功后通过 store action 触发列表刷新
- 通过路由参数、查询参数或返回标记控制页面刷新
- 组件间短链路通信优先使用 `props` 和 `emit`
- 同一模块内的跨页数据共享优先通过模块 store 处理

推荐的刷新规则如下：

- 新建/编辑/发布成功后返回列表页，并触发列表重新拉取
- 批改或结果处理完成后刷新详情页与结果汇总区
- 工作台统计数据由公共 store 统一刷新，不在每个业务页手写重复逻辑

### 7.5 通用交互状态规范

所有业务页都应统一处理以下界面状态：

- 首次加载态
- 空数据态
- 请求失败态
- 提交中态
- 操作成功反馈态

其中：

- 列表页优先提供筛选、分页、空态和重试
- 详情页优先提供返回关系、主操作按钮和错误反馈
- 处理页优先提供草稿保护、提交确认和完成反馈

## 8. 当前脚手架到目标结构的迁移

## 10. 落地约束与结论

### 10.1 落地约束

后续实现必须遵守以下约束：

- 业务页面必须落到 `src/views/{module}/`
- 业务组件必须落到 `src/components/{module}/`
- 业务 store 必须落到 `src/stores/{module}/`
- 路由文件统一放在 `src/router/`
- 教师端菜单必须由后台子路由元数据驱动
- 所有业务页必须配置 `meta.title`
- `src/api/*` 视为生成代码，不作为手写业务层
- 接口统一通过 `src/request.ts` 和生成函数访问

### 10.2 文档结论

新系统前端不应继续沿用旧系统“按角色平铺菜单”的页面思路，而应基于当前新项目已形成的 Vue 前端骨架，构建“统一壳层 + 模块归属 + 页面层级 + 状态分层”的逻辑结构。

在实现上，学生端和教师端是产品入口维度，`user`、`homework`、`exam`、`experiment`、`common` 是代码组织维度；路由是导航的唯一来源，布局负责承载，store 负责跨页状态，生成 API 负责接口边界。这样既能继承当前项目的新技术架构，也能把旧系统业务逐步迁移为可持续维护的新前端系统。
